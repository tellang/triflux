## Problem Statement

triflux TUI 대시보드(tui.mjs, 1376줄)는 로컬 headless 워커만 표시한다. remote-spawn으로 ultra4에 Claude/Codex 세션을 spawn하면 상태가 완전히 블라인드. 오늘 ultra4에서 codex-swarm 5개를 돌렸는데 전멸한 줄도 몰랐고, 좀비 node 프로세스 28개가 2.3GB RAM을 먹고 있었다. psmux 세션이 죽을 때 child process cleanup이 없어서 codex/gemini npx 프로세스가 고아로 남는다.

## What Makes This Cool

k9s 스타일 터미널 대시보드에서 로컬+원격 에이전트를 하나의 뷰로 본다. 워커 선택하고 Enter 치면 바로 그 세션에 붙는다. 원격 SSH 세션이든 로컬 psmux든 동일 인터랙션. 완료되면 Windows toast가 뜨고, 좀비가 쌓이면 자동으로 죽인다. 분산 에이전트 메시의 제어판.

## Constraints

- Node.js ESM (기존 triflux 스택)
- Zero-dep ANSI Direct (v7.1 아키텍처 결정, 외부 TUI 프레임워크 안 씀)
- Windows + psmux + WT 환경
- 기존 tui.mjs의 altScreen, rawMode, virtual row buffer, Catppuccin Mocha 테마 유지
- Lake 3e 모듈 소비: conductor.mjs, remote-probe.mjs, notify.mjs, remote-watcher.mjs

## Premises

1. TUI의 #1 문제는 "원격 블라인드 스팟" — 원격 세션 상태가 아예 안 보인다.
2. 인터랙션 모델은 k9s 패턴 — 워커 리스트에서 Enter 치면 해당 세션에 바로 붙는다.
3. 기존 Lake 3e 코드 재사용 — conductor, remote-probe, notify, remote-watcher 전부 구현 완료.
4. 완료/실패 시 자동 알림은 필수 — 폴링으로 눈 확인은 실패 모드.
5. Zero-dep ANSI Direct 유지.

## Approaches Considered

### Approach A: Surgical TUI Extension
기존 tui.mjs에 Enter→attach + 원격 카드 + notify 연동만 추가. 빠르지만 관심사 혼합.

### Approach B: Dashboard Controller Layer (Selected)
tui-remote-adapter.mjs로 원격 상태를 로컬 워커와 동일 인터페이스로 통합. TUI는 로컬/원격 구분 없이 같은 데이터 렌더링. 좀비 cleanup 독립 모듈.

### Approach C: Full Agent Mesh Dashboard
Hub MCP 브릿지 기반 실시간 양방향. Ocean 스코프, Hub 의존성 높음. Lake 5에서 재고.

## Recommended Approach: B — Dashboard Controller Layer

### 신규 파일

#### 1. `hub/team/tui-remote-adapter.mjs` (~300줄)

원격 세션을 TUI가 이해하는 로컬 워커 형식으로 변환하는 어댑터.

```
export function createRemoteAdapter(opts)
  → { start, stop, getWorkers, on, off }

이벤트:
  - workerUpdate: {name, cli, status, host, snapshot, ...}
  - workerCompleted: {name, host, exitCode}
  - workerFailed: {name, host, reason}
  - workerInputWait: {name, host, pattern}
```

**동작:**
- remote-watcher.mjs를 소비하여 tfx-spawn-* 세션 폴링
- conductor.mjs의 getSnapshot()에서 remote session 상태 수집
- 폴링 결과를 `tui.updateWorker()` 형식으로 변환
- host 필드 추가 (ultra4, m2, local 등)
- 완료/실패 시 notify.mjs의 createNotifier().notify() 자동 호출

**워커 데이터 구조 (tui.mjs와 호환):**
```javascript
{
  cli: "claude",          // 또는 codex, gemini
  role: "swarm-task-1",   // conductor config.id 또는 watcher sessionName에서 파생
  status: "running",      // pending, running, completed, failed (STATES→status 변환 아래 참조)
  host: "ultra4",         // 로컬이면 "local"
  remote: true,
  sshUser: "SSAFY",       // references/hosts.json의 ssh_user 필드에서 로드
  sessionName: "tfx-spawn-ultra4-26e94d8f",  // conductor session.id 또는 watcher sessionName
  snapshot: "최근 출력 마지막 5줄...",  // remote-watcher의 lastOutput (capture-pane 결과)
  conductor: {            // conductor 상태 머신 정보
    state: "healthy",     // STATES enum 값
    restarts: 0,
    probeLevel: "L1",     // 마지막 probe 레벨
  },
}
```

**STATES → TUI status 변환 테이블:**
| conductor STATES | TUI status |
|-----------------|------------|
| INIT, STARTING | "pending" |
| HEALTHY | "running" |
| STALLED, INPUT_WAIT, FAILED, RESTARTING | "running" (with conductor.state로 세부 표시) |
| COMPLETED | "completed" |
| DEAD | "failed" |

**데이터 소스 매핑:**
- `role`: conductor → `session.config.id`, watcher → sessionName 파싱 (tfx-spawn-{host}-{id})
- `snapshot`: remote-watcher의 `lastOutput` (capture-pane 텍스트). conductor getSnapshot()에는 없음.
- `sshUser`: `references/hosts.json` → `hosts[host].ssh_user`
- `sessionName`: conductor → `session.id`, watcher → `sessionName`
- conductor.state/restarts/probeLevel: conductor `getSnapshot()` → `{state, restarts, health}`

**adapter→notify 이벤트 변환:**
```javascript
// remote-watcher emits: sessionCompleted, sessionFailed, sessionInputWait
// adapter translates to notify event:
watcher.on("sessionCompleted", ({sessionName, exitCode}) => {
  notifier.notify({
    type: "completed",           // notify.mjs NOTIFY_EVENT_TYPES
    sessionId: sessionName,
    host: resolveHost(sessionName),
    summary: `exit ${exitCode}`,
  });
});
// sessionFailed → type: "failed", sessionInputWait → type: "inputWait"
```

**adapter 라이프사이클:**
- `createLogDashboard(opts)` 내부에서 `createRemoteAdapter(opts)` 생성
- dashboard `close()` → `adapter.stop()` → watcher/probe 정리
- adapter가 conductor `stateChange` 이벤트 구독 (primary), remote-watcher는 conductor 미등록 세션용 supplemental

**TUI paneName 키 컨벤션:**
- 로컬 워커: 기존 paneName (예: "worker-1")
- 원격 워커: `remote:{sessionName}` (예: "remote:tfx-spawn-ultra4-26e94d8f")

#### 2. `hub/team/process-cleanup.mjs` (~100줄)

고아 node/python 프로세스 감지 및 정리.

```
export function createProcessCleanup(opts)
  → { scan, kill, getOrphans }

export function findOrphanProcesses(opts)
  → [{pid, name, ramMB, parentPid, cmdLine, age}]
```

**동작:**
- `Get-Process` (Windows) 또는 `ps aux` (Unix)로 node/python 프로세스 스캔
- parent가 없거나 dead cmd인 프로세스 = 고아 후보
- `Get-CimInstance Win32_Process` (Windows) — parent PID + cmdLine 접근 필수
- 화이트리스트 (cmdLine 패턴 매칭):
  - `claude` (프로세스명) — 현재 세션
  - cmdLine에 `oh-my-claudecode` 포함 — OMC bridge MCP
  - cmdLine에 `triflux/hub/s` 포함 — hub server
  - `CCXProcess` parent — Adobe Creative Cloud
- `psmux list-sessions`와 교차 검증: 활성 세션 소속 PID는 제외
- kill 시 SIGTERM → 5s 대기 → SIGKILL
- **안전장치**: kill 전 대상 PID 목록을 stdout에 표시, TUI `c` 키는 확인 후 실행
- **추정 ~200줄** (Get-CimInstance 파싱 + psmux 교차검증 + 에러 핸들링)

**통합 지점:**
- conductor.mjs의 `shutdown()` — graceful shutdown 시 child process tree kill
- tui 대시보드 `?` 도움말에 "c: cleanup zombies" 키바인딩
- PreToolUse/PostToolUse 훅으로 세션 종료 시 자동 실행 가능

### tui.mjs 수정 (~150줄)

#### Enter 키 핸들러 (핵심)

```javascript
// handleInput 함수에 추가 (line ~960)
if (key === "\r" || key === "\n") {  // Enter
  if (!selectedWorker) return;
  const w = workers.get(selectedWorker);
  if (!w) return;

  if (w.remote && w.sessionName) {
    // 원격: SSH + psmux attach를 새 WT 탭에서 열기
    // exitAltScreen → wt.exe 실행 → enterAltScreen
    attachRemoteSession(w);
  } else if (w.paneName) {
    // 로컬: psmux attach를 새 WT 탭에서 열기
    attachLocalSession(w);
  }
  return;
}
```

**attachRemoteSession(worker):**
1. `input.pause()` + `input.setRawMode(false)` — rawMode 해제 (키 이벤트 차단)
2. `exitAltScreen()` — altScreen 임시 퇴장
3. `execFile("wt.exe", ["-w", "0", "nt", "--title", `${worker.host}:${worker.role}`, "--", "ssh", `${worker.sshUser}@${hostsJson.hosts[worker.host].tailscale.ip}`, "-t", `psmux attach -t ${worker.sessionName}`])` — 새 WT 탭에서 SSH attach (fire-and-forget, detached)
4. 200ms `setTimeout` — WT 탭 열림 대기
5. `enterAltScreen()` — 대시보드 복귀
6. `input.setRawMode(true)` + `input.resume()` — rawMode 재활성화
(패턴 참고: attach-fallback.mjs:20-27의 input pause/resume 흐름)

**sshUser 해결 순서:** worker.sshUser → hosts.json ssh_user → ~/.ssh/config Host 매칭 → 현재 user

**attachLocalSession(worker):**
1. `input.pause()` + `input.setRawMode(false)`
2. `exitAltScreen()`
3. `execFile("wt.exe", ["-w", "0", "nt", "--title", worker.role, "--", "psmux", "attach", "-t", worker.sessionName])`
4. 200ms `setTimeout`
5. `enterAltScreen()`
6. `input.setRawMode(true)` + `input.resume()`

**wt.exe 미설치 시:** `psmux attach -t {sessionName}`을 foreground 실행 (altScreen 완전 퇴장, attach 종료 후 복귀)

#### 자동 알림 연동

```javascript
// createLogDashboard에 onWorkerComplete 콜백 추가
// adapter가 workerCompleted 이벤트를 emit하면:
// 1. notify.mjs로 bell + toast 발송
// 2. TUI 상단에 1줄 flash 메시지: "✓ ultra4:task-1 completed (exit 0)"
// 3. 5초 후 자동 소멸
```

#### 새 키바인딩

| 키 | 동작 |
|----|------|
| Enter | 선택된 워커 세션에 attach (새 WT 탭) |
| c | process-cleanup 실행 (좀비 스캔+킬) |
| p | 원격 프로브 강제 실행 (즉시 폴링) |
| s | 선택된 워커에 프롬프트 전송 — 로컬: conductor.sendInput(), 원격: `psmux send-keys` over SSH (별도 코드 경로). v8 스코프에서는 로컬만. 원격 send는 Lake 3e+ 별도 설계. |

### conductor.mjs 수정 (~30줄)

#### 원격 세션 cleanup만 추가 (로컬 tree kill은 이미 구현됨)

**이미 있는 것:** `killSession()` → `cleanupChild()` → `forceKill(pid)` → `killProcess(pid, {signal:'SIGKILL', tree:true, force:true})`. 로컬 세션의 process tree kill은 conductor.mjs:122-125에서 이미 동작.

**추가할 것 (원격 세션만):**
```javascript
// killSession() 분기:
if (session.config.remote) {
  // SSH 경유 psmux kill-session
  const host = session.config.host;
  const sshUser = hostsJson.hosts[host]?.ssh_user;
  const ip = hostsJson.hosts[host]?.tailscale?.ip;
  execFile("ssh", [`${sshUser}@${ip}`, "psmux", "kill-session", "-t", session.id],
    { timeout: 10_000 });
}

// shutdown() 에서:
// 기존 로컬 SIGTERM→SIGKILL 흐름 유지
// 원격 세션은 위 SSH kill-session으로 정리
```

## Open Questions (Resolved)

1. ~~원격 폴링 간격: 10초 vs 5초?~~ → **10초 유지.** 실시간 아닌 near-realtime으로 충분. SSH 연결 비용 고려.
2. ~~좀비 cleanup을 훅으로 자동 실행할지?~~ → **둘 다.** c 키 수동 + conductor shutdown 시 자동. 훅은 v8+에서 검토.
3. ~~Enter→attach 후 WT 탭 닫히면?~~ → **영향 없음.** 새 WT 탭은 독립 프로세스. 대시보드는 200ms 후 altScreen 복귀. WT 탭 닫혀도 TUI에 영향 없음.
4. ~~s 키 원격 프롬프트 전송 UI?~~ → **v8 스코프에서는 로컬만.** 원격 send-keys는 별도 설계 필요.
3. Enter→attach 후 WT 탭이 닫히면 다시 대시보드로 돌아오는 UX?
4. 원격 세션에 프롬프트 전송(s 키)할 때 입력 UI 어떻게?

## Success Criteria

1. ultra4에서 codex-swarm 돌리면 TUI에 실시간 상태 표시
2. 워커 완료 시 Windows toast 자동 알림
3. Enter 키로 로컬/원격 세션 attach 가능
4. psmux 세션 종료 시 좀비 node 0개
5. 기존 로컬-only TUI 기능 100% backward compatible

## What I noticed about how you think

- "그놈의 좀비좀 없었으면 해" — 추상적 기능이 아니라 오늘 직접 겪은 고통에서 요구사항이 나왔다. 이게 진짜 제품 감각.
- ultra4 스웜 5개 전멸 + 좀비 2.3GB를 발견했을 때, "왜 안 보여?"가 첫 반응이었다. 문제를 시스템의 결함으로 귀속시키는 건 엔지니어링 마인드.
- k9s를 레퍼런스로 잡은 건 좋은 선택. 터미널 파워 유저가 쓰는 도구의 인터랙션 패턴을 가져오는 거니까.
