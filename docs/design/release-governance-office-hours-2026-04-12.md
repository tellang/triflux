## Problem Statement
triflux는 이미 npm 패키지, GitHub 저장소, Claude Code marketplace 메타데이터를 동시에 운영하는 제품인데, 릴리즈 계약은 아직 maintainer의 기억에 많이 의존한다.
현재 저장소에는 `.github/`가 없어서 브랜치 규칙, 이슈/PR 입력 형식, 릴리즈 체크리스트가 코드베이스 밖에 존재한다. 동시에 버전은 `package.json` 10.9.17, `.claude-plugin/marketplace.json` 10.9.17, `.claude-plugin/plugin.json` 10.3.4, `packages/triflux/package.json` 10.9.14, `package-lock.json` 10.9.14로 드리프트가 발생해 있다.

## What Makes This Cool
가장 멋진 버전은 세 가지다. 브랜치 진입 규칙이 곧 PR 품질을 결정하고, 버전 하나가 npm/GitHub Release/Claude marketplace까지 전파되며, Claude Code가 다음 턴에 그대로 실행 가능한 PRD를 바로 먹는 구조다.
이렇게 되면 triflux의 운영 체계 자체가 triflux다운 제품 데모가 된다. 멀티 에이전트 오케스트레이터가 자기 릴리즈도 오케스트레이션하는 셈이다.

## Constraints
- 새 런타임 의존성 없이 설계해야 한다. 현재 저장소 원칙도 `No new dependencies without explicit request.`다.
- 기존 배포 표면은 유지해야 한다. npm 패키지 이름 `triflux`, GitHub repo `tellang/triflux`, Claude marketplace source=`npm`은 깨지면 안 된다.
- 단일 maintainer + 다중 에이전트(worktree, swarm, remote) 환경에 맞아야 한다. 즉, trunk는 단순해야 하지만 기계 검증은 강해야 한다.
- 이번 턴은 구현이 아니라 문서화와 초안 작성까지만 한다.

## Premises
1. triflux에는 “브랜치 전략 / 릴리즈-버저닝 / 배포 fan-out”을 하나의 운영 시스템으로 묶는 문서가 필요하다.
2. 버전의 single source of truth는 루트 `package.json`이어야 하고, 나머지 아티팩트는 생성 또는 동기화 대상으로 내려가야 한다.
3. npm publish, GitHub Release, Claude marketplace 반영은 별도 의식이 아니라 하나의 release transaction으로 다뤄야 한다.
4. `.github/` 부재는 지금 규모에서 오히려 리스크다. 이슈/PR 입력 규격이 없으면 multi-agent 변경이 점점 더 리뷰 불가능해진다.

## Approaches Considered
### Approach A: 문서 우선, 수동 체크리스트 고정
- Summary: 정책 문서와 체크리스트만 먼저 만든다. 구현은 나중에 수동으로 따른다.
- Effort: S
- Risk: Low
- Pros:
  - 이번 턴에 가장 빨리 끝난다.
  - 새 스크립트 없이도 maintainer 판단 기준을 즉시 통일할 수 있다.
  - 실패 비용이 거의 없다.
- Cons:
  - 버전 드리프트 같은 문제를 기계적으로 막지 못한다.
  - 사람이 바쁘면 다시 예전 습관으로 돌아간다.
  - Claude Code가 실행하기에는 모호한 부분이 남는다.
- Reuses:
  - existing docs/prd 템플릿
  - 현재 `scripts/pack.mjs`

### Approach B: 정책 + 생성용 PRD + draft artifact 세트
- Summary: 이번 턴에 Claude Code가 바로 이어받을 수 있는 PRD 3개와 draft 3개를 만든다. 구현은 후속 턴에서 문서 기준으로 생성한다.
- Effort: M
- Risk: Low-Med
- Pros:
  - 지금 필요한 산출물과 다음 구현 턴의 입력이 동시에 준비된다.
  - 브랜치/릴리즈/배포를 한 vocabulary로 정리할 수 있다.
  - 새 의존성 없이 스크립트 기반 자동화로 자연스럽게 이어진다.
- Cons:
  - 실제 enforcement는 아직 없다.
  - draft를 실제 `.github/`나 `scripts/release/`로 옮기는 구현 턴이 추가로 필요하다.
- Reuses:
  - root/package/plugin 메타데이터 구조
  - `scripts/pack.mjs`
  - archived PRD의 release shard 관례

### Approach C: release-please/changesets 급 전체 표준화
- Summary: 외부 릴리즈 프레임워크를 도입해서 버전 범프, changelog, publish를 전면 자동화한다.
- Effort: L
- Risk: Med-High
- Pros:
  - long-term automation은 가장 강하다.
  - GitHub-native release flow와 잘 맞는다.
- Cons:
  - 현재 저장소의 수동 pack/workspace 구조와 충돌 가능성이 높다.
  - 새 의존성, 새 학습비용, 기존 문서/스크립트 재조정이 필요하다.
  - 지금 필요한 건 “정렬된 계약”인데, 이건 도구 도입부터 시작해서 순서가 거꾸로다.
- Reuses:
  - 거의 없음. 현재 스크립트 체계와 맞물리도록 다시 짜야 한다.

## Recommended Approach
Approach B를 추천한다.
이유는 단순하다. 지금 triflux에 없는 건 automation 툴이 아니라 운영 계약이다. 계약이 없는데 automation부터 넣으면 예외만 자동화된다. 먼저 Claude Code가 바로 구현할 수 있는 PRD와 초안으로 계약을 고정하고, 다음 턴에서 `.github/`와 `scripts/release/`를 생성하는 게 맞다.

## Open Questions
- stable만 운영할지, `canary` dist-tag를 공식 채널로 열지.
- GitHub Release note를 수동 편집 허용할지, 템플릿 강제만 둘지.
- Claude marketplace 반영을 release transaction 안에서 파일 sync + 검증까지만 할지, publish verification까지 포함할지.

## Success Criteria
- 브랜치/이슈/PR 거버넌스를 정의한 PRD 1개와 draft 1개가 있다.
- 릴리즈/버저닝 정책을 정의한 PRD 1개와 draft 1개가 있다.
- npm/GitHub Release/Claude marketplace 배포를 하나의 transaction으로 묶는 PRD 1개와 draft 1개가 있다.
- 각 PRD는 실제 생성 대상 파일, 검증 명령, 완료 조건을 포함한다.
- 각 draft는 후속 턴에서 거의 그대로 `.github/`, `docs/process/`, `scripts/release/` 초안으로 옮길 수 있다.

## Distribution Plan
이번 산출물은 repo 문서로 배포된다.
후속 구현 턴에서는 `scripts/release/check-sync.mjs` → `scripts/release/prepare.mjs` → `scripts/release/publish.mjs` 체인과 `.github/workflows/release.yml`을 만들어, 한 release transaction에서 npm publish, git tag/GitHub Release, marketplace metadata sync를 처리하도록 설계한다.

## What I noticed about how you think
- 당신은 그냥 “릴리즈 자동화 해줘”라고 하지 않고, “브랜치 전략 이슈 및 pr 양식”, “릴리즈 전략 버저닝 전략”, “npm 배포 깃 릴리즈 클로드 코드 마켓 배포”를 따로 찢어서 말했다. 이건 문제를 surface별로 나눠 보는 사람의 말이다.
- “이 세가지를 좀 통일해서”라는 표현이 핵심이었다. 기능 추가보다 운영 계약 정렬이 먼저라는 감각이 있다.
- “각각 필요한 범위의 클로드 코드 생성을 위한 prd 및 초안”이라고 한 것도 좋다. 구현보다 생성 입력의 품질을 먼저 챙기고 있다.
